# TECHNICAL_PLAN — aiGitWorktreeManager (gwm)

Детальный технический план. Терминология «Этап N» — по аналогии с соседними проектами портфеля (`orm-nplus1-radar`, `ai-knowledge-base`). Этап 0 = bootstrap.

---

## 1. Выбор TUI-технологии (обоснование)

Требование из идеи: инструмент **лёгкий, для повседневного терминального использования**, не тяжёлое GUI. Рассмотрены три варианта на Kotlin:

| Вариант | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| **Kotlin/Native CLI** | Мгновенный старт (нет JVM), маленький бинарь | Незрелый тулинг для TUI, нет готовых терминальных библиотек уровня Mordant, сложнее кросс-платформенная сборка, ещё одна «мышца», но не та, что просили качать (JVM-экосистема — основная в портфеле) | ❌ |
| **Compose for Desktop** («текстовый UI») | Богатый декларативный UI | Это **графический** тулкит (Skia/окно), тянет тяжёлый рантайм, не работает по SSH/в чистом терминале — прямо противоречит требованию «лёгкий терминальный инструмент» | ❌ |
| **JVM CLI + терминальный тулкит (Mordant + Clikt)** | Быстрый старт для JVM-утилиты, работает в любом терминале и по SSH, зрелые pure-JVM библиотеки, знакомая экосистема (Gradle/Kotlin, как в остальном портфеле), `application`-плагин даёт готовый запускаемый дистрибутив | Требует JVM в рантайме (у пользователя JDK уже есть) | ✅ **выбрано** |

**Итог: обычный JVM CLI с текстовым TUI поверх [Mordant](https://github.com/ajalt/mordant) (рендеринг таблиц, цвета, интерактивный выбор) и [Clikt](https://github.com/ajalt/clikt) (разбор аргументов/подкоманд).** Обе библиотеки — от одного автора (Ajalt), pure-JVM, лёгкие, без нативных зависимостей. Это ровно та «мышца», которую хотел прокачать пользователь: desktop-CLI как отдельный класс задач относительно IntelliJ-плагина (`orm-nplus1-radar`) и Ktor-бэкенда (`aiProjectDashboard`).

**Почему не «настоящий» full-screen ncurses-подобный TUI (alternate screen, произвольное позиционирование):** для MVP хватает line-oriented интерактивности Mordant (таблицы + `interactiveSelectList`/prompt). Full-screen перерисовка — кандидат в POST_MVP, если понадобится живой авто-refresh. Не усложнять раньше времени.

### Стартовая задержка JVM

Холодный старт JVM (~0.3–0.5 с) для повседневной утилиты приемлем и не блокирует. Если когда-нибудь станет мешать — путь ускорения зафиксирован в POST_MVP (AppCVDS / GraalVM native-image), но это не входит в MVP и не влияет на архитектуру.

## 2. Стек

- **Язык:** Kotlin (JVM), toolchain **JDK 17** (установлен через SDKMAN; `source ~/.sdkman/bin/sdkman-init.sh`).
- **Сборка:** Gradle (wrapper 8.12), плагины `kotlin("jvm")` + `application`. `application { mainClass = "dev.alkom.gwm.MainKt"; applicationName = "gwm" }` — `./gradlew installDist` даёт запускаемый скрипт `build/install/gwm/bin/gwm`.
- **Зависимости:** `com.github.ajalt.mordant:mordant` (TUI-рендеринг), `com.github.ajalt.clikt:clikt` (CLI), `kotlin("test")` для тестов (JUnit Platform).
- **Package root:** `dev.alkom.gwm`.
- **Git под капотом:** внешний `git` через `ProcessBuilder` — не JGit (см. §4).

## 3. Архитектура

Тонкие, независимо тестируемые слои. Чистая логика (парсинг) отделена от I/O (процессы, терминал), чтобы её можно было покрыть юнит-тестами без git и без TTY.

```
src/main/kotlin/dev/alkom/gwm/
├── Main.kt                 # Clikt entrypoint + подкоманды (list, ...) — тонкий слой UI
├── git/
│   ├── GitCommand.kt       # ProcessBuilder-обёртка: git с рабочей директорией + таймаутом
│   ├── Worktree.kt         # доменная модель одного worktree
│   ├── WorktreeParser.kt   # ЧИСТАЯ функция: парсит `git worktree list --porcelain`
│   └── WorktreeService.kt  # операции над одним репо: list/withDirtyFlags/add/remove/prune
├── scan/                   # (Фаза 2) обход корня репозиториев, агрегация, orphaned-эвристики
└── ui/                     # (Этап 1+) интерактивные экраны Mordant (выбор строки, диалоги)
```

**Ключевой принцип:** `WorktreeParser` не знает ни про процессы, ни про терминал — на входе строка `--porcelain`, на выходе `List<Worktree>`. Это самая хрупкая и самая важная логика, и она полностью покрывается юнит-тестами (`WorktreeParserTest`). `GitCommand`/`WorktreeService` — тонкие обёртки над реальным git; для них — интеграционные тесты на временном репозитории (появятся на Этапе 1+).

## 4. Почему внешний `git` через `ProcessBuilder`, а не JGit

- **Точное соответствие семантике worktree.** Locks, `prune`, `prunable`-флаги, форматы `--porcelain` — всё берётся у **установленного у пользователя** git; не нужно догонять поведение библиотеки.
- **Лёгкость.** Не тянем крупную зависимость (JGit) в инструмент, вся ценность которого — быть лёгким.
- **Прозрачность.** Любая операция — это ровно тот `git`-вызов, который пользователь мог бы набрать сам; при ошибке показываем его stderr дословно.

Дисциплина обёртки (реализована в `GitCommand`): всегда явная рабочая директория, всегда таймаут (дефолт 30 с) с `destroyForcibly()` — зависший git не должен фризить TUI; stdout/stderr разделены; `--porcelain`/`-z`-форматы предпочтительнее человекочитаемых (стабильный парсинг). Подробные конвенции — в `.claude/skills/kotlin-worktree-tui-dev/SKILL.md`.

## 5. Переключение cwd (осознанное ограничение и его обход)

Дочерний процесс **не может** сменить рабочую директорию родительского шелла — это фундаментальное свойство POSIX, не баг. Поэтому `gwm` не «переносит» пользователя в worktree сам. Два уровня решения:

1. **Базовый (Этап 6, всегда работает):** `gwm` печатает готовую строку `cd '<path>'` выбранного worktree — скопировал, вставил, перешёл.
2. **Бесшовный (опциональный, ставится один раз):** shell-функция-обёртка в `.bashrc`/`.zshrc`, которая вызывает `gwm` в специальном режиме (печать *только* пути в stdout) и делает `cd` уже в родительском шелле:
   ```bash
   gwm() {
     if [ "$1" = "cd" ]; then
       local dir; dir="$(command gwm --print-path "${@:2}")" && [ -n "$dir" ] && cd "$dir"
     else
       command gwm "$@"
     fi
   }
   ```
   Тогда `gwm cd <fuzzy>` реально меняет директорию. Установка функции — разовое действие пользователя, `gwm` её только предлагает (печатает готовый сниппет), но не лезет в dotfiles сам.

Это самый элегантный из доступных обходов: он честен насчёт ограничения ОС и не требует привилегий/хаков.

## 6. Почему нет docker-compose

`gwm` — **чисто локальный CLI/TUI-инструмент**. У него нет сервисного рантайма: нет HTTP-сервера, нет БД, нет фоновых воркеров, нет ничего, что имело бы смысл поднимать в контейнере. Единственная внешняя зависимость — установленный локально `git`, который и так есть в системе разработчика. Поэтому docker-compose в проекте **не используется и не планируется** — это была бы пустая обвязка. Стоимость эксплуатации инструмента — $0.

## 7. Разбивка по Этапам (технически)

Все Этапы MVP (0–6) реализованы, покрыты тестами и смержены в `master`. Ниже — что именно сделано на каждом, с точкой входа в код.

- **Этап 0 (сделано ✅):** Gradle-скелет, `application`-плагин, `GitCommand`/`Worktree`/`WorktreeParser`/`WorktreeService`, команда `gwm list <repo>` с Mordant-таблицей, юнит-тесты парсера. `./gradlew build` зелёный.
- **Этап 1 (сделано ✅):** интерактивный список (Mordant `interactiveSelectList`), навигация стрелками, выбор worktree; экраны вынесены в `ui/InteractiveScreen.kt` + `ui/WorktreeTable.kt`. Команда `gwm interactive`. Форматирование строк (`rowLabel`/`entryKey`) — чистые функции, покрыты `InteractiveScreenTest`.
- **Этап 2 (сделано ✅):** `gwm create <branch> [<path>]` с `--base <ref>` (дефолт `HEAD`) и `--repo`; дефолтный путь `../<repo>-<branch>` (`WorktreeService.defaultWorktreePath`), вызов `WorktreeService.add`. Интеграционные тесты на временном репо (`WorktreeServiceIntegrationTest`).
- **Этап 3 (сделано ✅):** `gwm remove <path|branch>` + пункт «удалить» в `interactive`: подтверждение (`YesNoPrompt`), обработка dirty (`safeRemove` → `BLOCKED_DIRTY`, `--force` только по явному согласию), `WorktreeService.prune`.
- **Этап 4 (сделано ✅):** `scan/` — обход корня (`~/Projects/ai-projects/*`, конфигурируемо через `--root`/`GWM_ROOT`), поиск git-репо (`RepoScanner`, только `.git`-директория = primary checkout), параллельная (coroutines, `Dispatchers.IO`) агрегация worktree всех репо в одну таблицу с изоляцией ошибок (`RepoError`). Команда `gwm scan`.
- **Этап 5 (сделано ✅):** orphaned-эвристики (`git branch --merged`, отсутствие upstream, `prunable`) — чистый `OrphanClassifier` + факты из `WorktreeService.withOrphanStatus`; визуальные пометки в колонке «Orphaned» (`WorktreeTable`) и inline-бейдж в интерактивном списке.
- **Этап 6 (сделано ✅):** корневая опция `--print-path <fuzzy>` (не подкоманда — Clikt резервирует токены с ведущим дефисом под опции, см. `Gwm` в `Main.kt`), резолвинг через `scan/WorktreeMatcher` (exact > substring, ambiguous = ошибка), `gwm cd <fuzzy>` через shell-функцию из `gwm shell-init` (`ui/ShellInit.kt`).

Каждый Этап проходил workflow из `CLAUDE.md` (Opus план+код → Sonnet тесты → Opus ревью → до 3 итераций → PR в `master`).

## 8. Тестирование

- **Юнит (без git, без TTY):** `WorktreeParser` и вся чистая логика скоринга/эвристик orphaned — детерминированные таблицы вход/выход. Первый ярус.
- **Интеграционные (реальный git, временный репо):** `GitCommand`/`WorktreeService`/`scan` гоняются на созданном в `@TempDir` репозитории с реальными worktree — детерминированный фейк реального стека, а не мок ProcessBuilder. Появляются с Этапа 1. (Соответствует установке портфеля: тестам нужен интеграционный ярус, не только мок-юниты.)
- **Команды:** `./gradlew test` (юнит + интеграционные), `./gradlew build` (компиляция + тесты + дистрибутив).

## 9. Известные детали сборки

- `org.gradle.configuration-cache=false` в `gradle.properties` — отключён осознанно: в соседнем Kotlin-проекте портфеля (`orm-nplus1-radar`, Этап 0) configuration-cache давал ложные ошибки сериализации тасков. Для маленького CLI выигрыш незначителен, поэтому отключаем целый класс ложных падений. Включить обратно — только при измеренной необходимости и с проверкой зелёной сборки.
- Gradle wrapper — 8.12; toolchain — JDK 17.
- Первая реальная сборка Этапа 0 (`./gradlew build`) прошла с первого раза зелёной; команда `gwm list .` проверена на самом репозитории — таблица worktree рендерится корректно.
