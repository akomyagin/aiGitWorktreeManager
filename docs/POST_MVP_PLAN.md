# POST_MVP_PLAN — aiGitWorktreeManager (gwm)

Границы MVP и идеи «на потом». Ничего отсюда не реализуется без явного запроса — это roadmap, а не бэклог обязательств. MVP (Этапы 0–6) закрывает: обзор worktree по всем репозиториям, orphaned-детекцию, create/remove, переключение.

## Явно ЗА границами MVP

- **Автоудаление orphaned worktree.** `gwm` в MVP только помечает и подсказывает; он никогда не удаляет worktree без явного действия пользователя. Автоочистка (даже опциональная) — сознательно отложена, чтобы не потерять незакоммиченную работу.
- **Не-git VCS** (Mercurial, jj/Jujutsu). Только git worktree.
- **Windows.** MVP таргетирует Linux/macOS (POSIX-шелл для cwd-обёртки). Windows-поддержка — отдельный вопрос (PowerShell-функция вместо bash-обёртки).

## Идеи на будущее

### UX / TUI
- **Full-screen TUI с авто-refresh.** Живая перерисовка (alternate screen) с автообновлением статусов по таймеру/файловому watch. Требует более тяжёлого TUI-режима Mordant — оправдано, только если line-oriented обзора станет мало.
- **Fuzzy-поиск по worktree** (в духе `fzf`) прямо в интерактивном списке при большом числе репозиториев.
- **Пометка «сколько коммитов ahead/behind upstream»** в таблице обзора.
- **Возраст worktree** (последний коммит / последнее изменение файлов) для приоритизации уборки.

### Функциональность
- **Создание worktree из удалённой ветки** (`git worktree add --track -b ...`) одним действием, включая fetch.
- **Bulk-операции:** удалить все orphaned за раз (с обязательным подтверждением каждого dirty).
- **Интеграция с `aiProjectDashboard`** — экспорт обзора worktree как данных для дашборда проектов портфеля.
- **Хуки:** запускать команду после создания worktree (например, `direnv allow`, копирование `.env`).

### Производительность / упаковка
- **GraalVM native-image** — мгновенный старт без JVM, один бинарь. Оправдано, только если холодный старт JVM (~0.3–0.5 с) начнёт реально мешать в повседневном использовании.
- **AppCDS** — более лёгкий способ ускорить старт JVM без перехода на native.
- **Distribution:** публикация как SDKMAN-кандидат / Homebrew formula, если инструментом захотят пользоваться другие.

### Конфигурация
- Конфиг-файл (`~/.config/gwm/config.toml`): список корней сканирования, дефолтный шаблон пути новых worktree, цветовая схема.

## Границы, которые НЕ двигаем

- Никакой сети, никакого сервера, никакой телеметрии. Инструмент остаётся полностью локальным и офлайн.
- `gwm` не пишет в `.git` напрямую — всегда через настоящий `git`.
- Не редактирует dotfiles пользователя сам — только предлагает готовый сниппет shell-функции.
