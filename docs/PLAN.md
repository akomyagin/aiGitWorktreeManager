# PLAN — aiGitWorktreeManager (gwm)

> Личный pet-проект соло-разработчика. Цель — прокачать Kotlin на новом классе задач (desktop TUI/CLI, работа с процессами и терминалом), сделав при этом инструмент, которым сам пользуешься каждый день. Экономическая целесообразность необязательна: чистый локальный инструмент, стоимость эксплуатации ~$0, сервера нет.

## 1. Что это

`gwm` — TUI-менеджер `git worktree` для терминала. Даёт **единый обзор** worktree сразу по всем локальным репозиториям пользователя (по умолчанию — `~/Projects/ai-projects/*`): какие worktree существуют, на какой они ветке, чистое рабочее дерево или грязное, есть ли «осиротевшие» (orphaned/stale) worktree. Создание, удаление и переключение между worktree — прямо из TUI.

## 2. Зачем (проблема, которую решаем)

`git worktree` из коробки полностью функционален, но неудобен в UX:

- **Нет обзора «что у меня вообще есть».** `git worktree list` работает только внутри одного репозитория; чтобы увидеть картину по всем проектам, надо вручную обойти каждый.
- **Нужно помнить пути.** Worktree лежат в разных директориях; переключение — это `cd` по запомненному пути.
- **Легко забыть про orphaned worktree.** Ветку смержили и удалили, а worktree остался висеть — занимает место и путает. `git worktree prune` чистит только совсем битые записи, а не «ветка ушла в никуда».

`gwm` закрывает именно эти дыры UX, не пытаясь заменить сам `git worktree` — под капотом всё делает настоящий `git`.

## 3. Для кого

Автор и такие же соло-разработчики, кто держит **много локальных репозиториев** и активно пользуется worktree (например, чтобы параллельно вести несколько веток/фич без постоянного `stash`/`checkout`). Профиль автора: соло, интроверт, реализация с AI-ассистентом; вся коммуникация — асинхронная (GitHub Issues).

## 4. Философия

- **Обёртка, а не замена.** Все мутации (`add`/`remove`/`prune`) — это вызовы настоящего `git`. `gwm` ничего не пишет в `.git` напрямую. Если git-команда упала — показываем её stderr, не «глотаем».
- **Read-only по умолчанию безопасно.** Обзор и сканирование ничего не меняют. Разрушающие действия (удаление worktree) — только по явному подтверждению.
- **Лёгкость важнее богатства.** Это инструмент на каждый день в терминале, не GUI-приложение. Быстрый старт, минимум зависимостей, работает по SSH.

## 5. Осознанные ограничения

- **Переключение cwd.** Дочерний процесс не может сменить рабочую директорию родительского шелла. `gwm` не «телепортирует» пользователя в worktree; он печатает готовую команду `cd` для копирования, а для бесшовного перехода предлагает опциональную shell-функцию-обёртку в `.bashrc`/`.zshrc` (детали — TECHNICAL_PLAN §5).
- **Определение «orphaned».** «Ветка смержена/удалена» определяется эвристически (`git branch --merged`, отсутствие upstream, `prunable`-флаг git). Это подсказка для человека, а не автоудаление — `gwm` никогда не удаляет worktree сам.

## 6. Этапы

Терминология «Этап N» — по аналогии с соседними проектами портфеля. Этап 0 = bootstrap.

> **Статус: MVP готов ✅.** Все Этапы 0–6 (Фаза 1 + Фаза 2) реализованы, покрыты тестами и смержены в `master` (PR #1–#4). Команды `gwm`: `list`, `interactive`, `create`, `remove`, `scan`, `shell-init` + корневая опция `--print-path`.

### Этап 0 — Bootstrap ✅
Инициализация репозитория: документация (`PLAN`/`TECHNICAL_PLAN`/`POST_MVP_PLAN`), `CLAUDE.md`, skill, скелет Gradle-проекта, минимальный `Main.kt` с командой `list`, парсер `git worktree list --porcelain` + юнит-тесты. Сборка `./gradlew build` — зелёная.

### Фаза 1 — обзор и управление по одному репозиторию (ядро MVP)

| Этап | Статус | Кратко |
|---|---|---|
| Этап 1 | ✅ | Полный обзор одного репо: интерактивный TUI-список worktree (Mordant `interactiveSelectList`), ветка, статус чистое/грязное, выбор строки стрелками (`gwm interactive`, `ui/InteractiveScreen.kt`) |

## 7. Определение готовности MVP

MVP считается готовым, когда по всем репозиториям из настроенного корня можно за один запуск: увидеть все worktree с ветками и статусом, отличить orphaned от активных, создать/удалить worktree и получить готовую команду перехода — без ручного обхода репозиториев и без запоминания путей.
